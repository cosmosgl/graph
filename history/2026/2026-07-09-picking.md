# Picking: change-gated ID buffers + async readback (points & links)

**Commits:** aae7b51, 42a79a5

## Why

Hover/picking was O(n) per detection with a GPU→CPU stall, for both element kinds:

- **Points**: every hover ran a full-point GPU pass (each point through a vertex
  shader that tested the cursor against its radius) and then a **synchronous**
  `readPixels` on the result.
- **Links**: every detection re-rasterized **all** links into a full-screen index
  buffer (no change gating), ran a lookup pass to sample the cursor pixel, and
  synchronously read that back.

At high element counts (the 100k stress test hovering mid-simulation) the
per-detection cost and the pipeline stall dominate the frame.

The fix decouples pick cost from the element count and takes the readbacks off the
hot path. The point scheme already existed on the `feat/3d` branch (855a41d); it
could not be cherry-picked here because that commit is entangled with 3D
infrastructure this 2D-only branch doesn't have (`SPACE_3D` shader defines,
`space3dModule`, `store.is3D`, 3D `Hovered` positions). So it was **ported** — 3D
paths stripped, and this branch's two 2D hover-correctness fixes re-merged into the
new fill shader. Link picking then got the same treatment (gating + async readback),
reusing the same machinery.

## The scheme (three parts)

### 1. Screen-space ID buffers

**Points** — `fill-picking-buffer.vert`/`.frag` rasterize a picking buffer:
`[index, x, y]` per pixel (index `-1` = empty), at half screen resolution capped at
1536px, `rgba32float`. Each point draws its circular sprite at its projected
position, carrying its index. Hover then reads only a **9×9 window** under the
cursor and picks the valid candidate nearest the cursor
(`resolveNearestPickedPoint`) — so a pick is independent of the point count once the
buffer is filled. The greyed-then-highlighted two-pass priority (highlighted wins)
is unchanged from the old draw order.

The port preserves this branch's 2D correctness work in the fill shader (see
[nan-point-removal](2026-06-27-nan-point-removal.md)):

- the **exit-texture guard** — a faded/removed (`NaN`-position) point is skipped, so
  hover never lands on an absent point, and
- **`pointDefaultSize` NaN-size resolution** — a `NaN` size resolves to the config
  default, matching `draw-points.vert`.

**Links** — the pre-existing full-screen link index buffer (`linkIndexFbo`,
`[index, 0, 0, validity]` per pixel, rendered by the `renderMode = 1` branch of the
link draw shader) is kept, but the cursor lookup now reads its **1×1 cursor pixel
directly** (`resolvePickedLinkIndex` reproduces the old lookup shader's
`floor(mouse / screen × size)` sample and `alpha > 0 && index >= 0` validity — same
behavior, no added tolerance). The intermediate lookup pass —
`hovered-line-index.vert`/`.frag`, its 1×1 FBO, command, uniform store and quad
buffer — is deleted. The link buffer stays full-res (thin 1px links would fall
between texels at half-res), so its memory footprint is unchanged.

The old `find-hovered-point.vert`/`.frag`, the 2×2 `hoveredFbo`, and the
per-detection full readbacks are removed.

### 2. Async readback

`PickingReadback` (`src/modules/Points/picking-readback.ts`) reads a cursor window
into a `PIXEL_PACK_BUFFER` and collects it via `fenceSync`/`clientWaitSync` one or
more frames later (`Graph.resolvePendingPick` in the frame loop). No synchronous
`readPixels` on the hover path, so the pipeline never stalls; the result arriving a
frame or two late is imperceptible for hover. luma.gl v9's `Buffer.readAsync` calls
`getBufferSubData` immediately (no fence), so the helper drives the raw WebGL2
objects itself. Points and links each own an instance (9×9 window / 1×1 pixel); both
reads are issued in the same detection — point-first, and since fences signal in
submission order, a ready link result implies its point result is ready too.

Clicks, drag-starts and long-presses still take **synchronous** reads
(`pickPointSync` / `pickLinkSync`) because they need the result inside the event —
`findHoveredItem(true)` must have `store.hoveredPoint` populated before it returns
(d3-drag reads it). Points keep priority: the link result is discarded whenever a
point is hovered (and on the sync path the link read is skipped entirely).

A result that lands **after the pointer left the canvas** is discarded:
`pointerleave` already cleared hover, and re-applying it would leave it stuck. This
keeps parity with the old sync path, which only detected while the pointer was on
the canvas.

### 3. Change gating

Each buffer is re-rendered only when its stale flag is set
(`Points.isPickingBufferStale` / `Lines.isLinkIndexBufferStale`). A static scene
with a still cursor does zero picking work; hovering a static scene costs only the
async reads.

| Marked stale by | Where |
|---|---|
| point positions move | simulation tick, drag, interpolation, mouse repulsion (frame loop, `markPickingBuffersStale`) |
| point data | `updatePointStatus`, `updateSize`, image-size update, `initPrograms`, `updatePositions` (links via `Graph.create()`) |
| link data | `updatePointsBuffer`, `updateWidth`, `updateArrow`, `updateColor`, `updateLinkStatus`, `updateCurveLineGeometry` |
| animated transitions | `Lines.setTransitionProgress` (widths/positions move footprints; color alpha gates pickability) |
| view transform (zoom) | `updatePickingBufferStaleness` diffs `transformationMatrix4x4` |
| canvas resize | `resizeCanvas` (buffers are sized to the screen) |
| any config change | `updateStateFromConfig` (sizes, widths, arrows, curvature, emphasis all feed the passes) |
| hovered link change | `applyPickedLink` / `pointerleave` — the hovered link is drawn **wider** in the index pass (hover hysteresis), so the hover itself is an input to the buffer |

Two subtleties worth noting:

- The link index pass **discards fully transparent fragments**, so a link's color
  alpha gates its pickability — that's why `updateColor` and color-animating
  transitions mark the link buffer stale.
- `updateLinkIndexBuffer` is self-healing: it (re)creates the buffer via
  `updateLinkIndexFbo()` when missing or after a resize, so enabling link hovering
  late (via `setConfig`) works without an explicit init step.

## Notes

- **Code layout.** The pure, stateless helpers — buffer sizing, cursor-window math,
  the window scan (`resolveNearestPickedPoint`) and the link-pixel decode
  (`resolvePickedLinkIndex`) — live in `picking-utils.ts`, with the tunables in
  `picking-constants.ts` and the async-readback object in `picking-readback.ts`.
  Only the GPU-command lifecycle (buffer render, FBO/readback ownership) stays in
  `Points` / `Lines`; `Graph` holds just the hover orchestration
  (`processHoverResult`, `applyPickedPoint`, `applyPickedLink`).
- The point picking FBO keeps a `depth16unorm` attachment even though depth is inert
  in 2D (`depthCompare` stays `'always'`). It's kept so the 3D nearest-wins depth
  test can layer back cleanly if this branch later merges 3D.
- `resolveNearestPickedPoint` returns `position: [x, y]` — `store.hoveredPoint` /
  `Hovered` stay 2D here; the buffer's third channel is written `0`.
- Bare-removal fades still keep hover live (unchanged): a removal alone queues only a
  `Positions` transition and the exit-texture guard handles absence — no size/color
  transition pauses hover. Links with a currently-absent endpoint are dropped by the
  index pass, as before.
